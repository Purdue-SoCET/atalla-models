## 1. Architecture Overview

The pipeline compiles a standard PyTorch `nn.Module` down to Atalla assembly and executes it on the functional simulator. Three components are integrated:

```
PyTorch nn.Module
    тФВ
    тФЬтФА atalla-graph/        FX graph capture, op normalization, tiling, code generation
    тФВ
    тФЬтФА aihw-ppci-compiler/  AtallaC тЖТ assembly compiler; PRIMARY compilation path
    тФВ                       for GEMM, Conv, Linear, ReLU, Softmax kernels
    тФВ
    тФФтФА functional_sim/      Assembler, .in file builder, and cycle-accurate emulator
```

**Two execution paths exist:**


**Key distinction:** `build_compiler.py` produces the instruction section only. The data section (DRAM layout with config pointers at `0x3C`, input tensors, weight data) is generated by `atalla-graph`'s `DRAMWriter` in `c_emitter.py`/`dram_builder.py`. Both halves are combined by `render_testfile()` to produce the complete `.in` file.

**Path 1 (build_*.py) is the lowest truth.** These hand-written scripts bypass the compiler entirely, directly encoding tested instruction sequences with matching data sections. They serve as golden references for validating Path 2.

**Path 2 (C compiler) is the primary pipeline.** All graph-level kernels (conv, linear, matmul, relu, softmax) generate AtallaC via `atalla-graph/kernels/*.py`, compile through ppci, schedule via `build_compiler`, and execute on the emulator. `atalla-graph` owns both code generation AND data section generation.

**Why MaxPool / Add / Mul use NumPy:**

MaxPool has a working assembly kernel (`build_maxpool.py`) that operates on **single-channel 2D tiles** (one H├ЧW plane at a time). Wiring it into the multi-channel pipeline requires a per-channel loop with correct DRAM address arithmetic. This was deferred to keep initial integration simple. **This is a wiring gap, not a missing kernel.**

Add and Mul appear in `BasicModule` as bias addition (`out + bias`) and residual scaling (`0.5 * residual`). They are scalar-broadcast or elementwise operations that don't meaningfully benefit from hardware execution. AlexNet's bias addition is folded into Conv2d/Linear modules and does not produce standalone add/mul nodes.

## 2. Component Deep Dive: atalla-graph

**Purpose:** PyTorch frontend that captures a model as an FX graph, normalizes operations to Atalla primitives, plans tile decomposition, and generates AtallaC source for each layer.

### 2.1 `graph/fx_capture.py` тАФ Graph Capture & Op Normalization

**What it does:**
1. Converts model weights to `bfloat16` (the hardware's native format)
2. Calls `torch.fx.symbolic_trace(model)` to get a flat FX graph
3. Runs `ShapeProp` to propagate tensor shapes through every node
4. Tags every node with `node.meta["atalla_op"]` тАФ a normalized string identifying the Atalla primitive

**Op mapping tables:** Three dictionaries map PyTorch operations to Atalla primitives:

| Source | Example | Maps to |
|--------|---------|---------|
| `_OP_MAP` (call_function) | `torch.matmul`, `F.relu`, `F.conv2d` | `"matmul"`, `"relu"`, `"conv"` |
| `_METHOD_MAP` (call_method) | `.relu()`, `.flatten()`, `.add()` | `"relu"`, `"flatten"`, `"add"` |
| `_MODULE_MAP` (call_module) | `nn.Conv2d`, `nn.Linear`, `nn.MaxPool2d` | `"conv"`, `"linear"`, `"maxpool"` |

Special handling:
- `addmm` (PyTorch's fused bias+matmul) is detected by name and mapped to `"linear"`
- `operator.add` and `operator.mul` are captured (these come from Python `+` and `*` operators in the forward pass)

**Limitation:** `symbolic_trace` cannot handle dynamic control flow. All models must be FX-traceable (no data-dependent branches, no dynamic shapes).

### 2.2 `graph/remove_ops.py` тАФ Graph Transforms

Three passes clean up the graph before code generation:

1. **BatchNorm folding** (`_fold_bn_into_conv`): Folds BN parameters into the preceding Conv2d's weight and bias using the standard formula: `W_new = W * gamma / sqrt(var + eps)`, `b_new = beta + (b - mu) * gamma / sqrt(var + eps)`. The BN node is then erased from the graph.

2. **Dropout removal** (`_remove_dropout`): Elides dropout nodes (inference mode) by replacing them with their input.

3. **Adaptive avg pool tagging** (`_remove_adaptive_avg_pool`): Tags `AdaptiveAvgPool2d(1,1)` nodes so they can be handled as NumPy fallback.

### 2.3 `graph/tile_planner.py` тАФ Tile Planning & DRAM Layout

**Hardware constraints:**
- Vector length (VL) = 32
- Systolic tile = 32├Ч32
- Scratchpad = 32 slots per bank
- Data format = bfloat16 (2 bytes per element)

**Per-op tile planning:**

| Op | Planning Logic |
|----|---------------|
| **Conv** | Extract H, W, C_in, C_out, R, S, stride, pad тЖТ compute Ho, Wo тЖТ M = Ho├ЧWo, K = R├ЧS├ЧC_in, N = C_out тЖТ tile counts = ceil(dim/32) |
| **Linear** | M=1, K=in_features, N=out_features |
| **Matmul** | M, K, N from tensor shapes, tile counts = ceil(dim/32) |
| **ReLU** | total_elements from input shape, width = min(total, 32) |
| **Softmax** | length from last dimension of input |
| **MaxPool** | H, W, C, pool_size, stride тЖТ H_out, W_out |
| **Add/Mul** | total_elements from output shape |
| **Flatten** | Passthrough, no tiling needed |

**DRAM address assignment:** Addresses are assigned sequentially starting at `0x1000`, aligned to 2KB tile boundaries. Each node gets `dram_addr` and `dram_bytes` in its metadata. Address `0x00`тАУ`0x3C` is reserved for the ADDR_TABLE (parameter passing to kernels).

### 2.4 `codegen/c_emitter.py` тАФ Code Generation (Primary)

**What it does:** For each FX node, generates either:
1. **AtallaC source** (for conv, linear, matmul, relu, softmax) тАФ compiled through ppci тЖТ `build_compiler.compile_asm` тЖТ emulator
2. **NumPy fallback** (for maxpool, add, mul, adaptive_avg_pool)

**Core dispatch function:**

```python
def emit_node(node, gm, activation_cache) -> LayerEmission:
    if atalla_op == "conv":     return emit_conv(...)    # sets em.c_source
    elif atalla_op == "linear": return emit_linear(...)   # sets em.c_source
    elif atalla_op == "relu":   return emit_relu(...)     # sets em.c_source
    elif atalla_op == "softmax":return emit_softmax(...)  # sets em.c_source
    elif atalla_op == "matmul": return emit_matmul(...)   # sets em.c_source
    elif atalla_op == "maxpool":return emit_maxpool(...)  # skip_emulator=True
    elif atalla_op == "add":    return emit_add(...)      # skip_emulator=True
    elif atalla_op == "mul":    return emit_mul(...)      # skip_emulator=True
```

**`LayerEmission` dataclass:**

| Field | Purpose |
|-------|---------|
| `c_source` | AtallaC source code for compilation |
| `instr_text` | Compiled instruction text in `.in` format (populated by `compile_and_assemble`) |
| `dram` | `DRAMWriter` with serialized weight/activation data |
| `output_addr` | DRAM address where output will be written |
| `output_shape` | Shape of the output tensor |
| `output_elements` | Number of bf16 elements in output |
| `skip_emulator` | If True, use `numpy_result` instead of running emulator |
| `numpy_result` | NumPy array for ops that skip the emulator |

**AtallaC code generators (in `atalla-graph/kernels/`):**

| Generator | Used by | Key technique |
|-----------|---------|---------------|
| `gemm.py` / `gemm_c(M, N, K)` | conv, linear, matmul | Tiled GEMM with `scpad_load/store`, `vector_load/store`, `load_weights`, `gemm()` |
| `relu.py` / `relu_c(total, width)` | relu | `vector_load` тЖТ `make_mask("<", v, zero)` + `vec_op_masked("*", v, 0.0, mask)` тЖТ `vector_store` |
| `softmax.py` / `softmax_c(length)` | softmax | `RMAX` тЖТ shift тЖТ `EXP` тЖТ `RSUM` тЖТ `1.0 / sum` (compiles to rcp_bf+mul_bf) |
| `maxpool.py` / `maxpool_c(...)` | maxpool | `vector_load` тЖТ `make_mask(">")` тЖТ blend тЖТ `vector_store` (vertical max only) |
| `common.py` | all | `sdma_ctl_val()`, `sdma_ctl_expr()` helpers |

All generators now use first-class intrinsics (`scpad_load/store`, `vector_load/store`, `load_weights`) instead of inline assembly for scratchpad and vector register operations. Only `lw_s`, `li_s`, and `halt` remain as inline asm.

**SDMA control register packing:** The `_sdma_ctl_val(sid, num_rows, num_cols, full_cols)` function packs scratchpad DMA parameters into a 32-bit control word: `(sid << 30) | ((num_rows-1) << 25) | ((num_cols-1) << 20) | (full_cols-1)`. The compiler cannot handle large constant stores, so `_sdma_ctl_expr()` uses inline asm `li_s` which `build_compiler` expands to `lui_s` + `addi_s`.

**Conv emission (`emit_conv`) pipeline:**
1. Extract Conv2d parameters (C_in, C_out, kernel size, stride, padding)
2. Transform input to NHWC layout, run `im2col()` тЖТ `(Ho*Wo, R*S*C_in)` matrix
3. Reshape weight to `(R*S*C_in, C_out)` matrix
4. Assign DRAM addresses: A_GMEM, W_GMEM, C_GMEM (sequentially, 4KB aligned)
5. Write ADDR_TABLE at offset 60: pointers, M, N, K, tile counts, tile size
6. Write input, weight, zero output to DRAM
7. Generate AtallaC via `_gemm_c(M, N, K)`

### 2.5 `codegen/asm_converter.py` тАФ Notation Bridge (DEPRECATED)

Previously converted ppci's assembly notation to the emulator's format. **Now superseded by `build_compiler.compile_asm()`**, which handles notation conversion, VLIW scheduling, and encoding in a single pipeline step. The converter is retained in the codebase but no longer called by the active pipeline.

`build_compiler.py` handles these notation transforms internally:

| Compiler Output | Internal Handling |
|----------------|-------------------|
| `add_s x1, x2, x3` | Mnemonic alias (`add_s` тЖТ `add.s`), register parsing (`xN` тЖТ register index) |
| `nop` / `halt` | Alias to `nop.s` / `halt.s` |
| `.section .text` | Directives stripped during parse |

### 2.6 `codegen/dram_builder.py` тАФ Tensor Serialization

Provides `extract_input_data()` which runs the actual PyTorch model (interpreter-style, node by node) to capture intermediate activations as reference values for validation. Also provides `bf16_from_float()` / `float_from_bf16()` conversion utilities.

### 2.7 `run_model.py` тАФ Orchestrator

End-to-end script that ties everything together:

1. Accepts `--model basic|alexnet` and `--scale` arguments
2. Calls `capture()` тЖТ `remove_ops()` тЖТ `plan_tiles()`
3. Populates `activation_cache` with placeholder/get_attr values
4. For each compute node in topological order:
   - Calls `emit_node()` to get a `LayerEmission`
   - If `skip_emulator`: stores `numpy_result` in `activation_cache`
   - Otherwise: calls `compile_and_assemble()` (compiles C source), writes `.in` file, runs emulator, reads bf16 output from emulator memory, stores in `activation_cache`
5. Collects per-kernel metrics (range, cosine sim, cycles, instructions, GEMM ops)
6. Compares final output against PyTorch reference using cosine similarity

## 3. Component Deep Dive: aihw-ppci-compiler

**Purpose:** A C compiler for the Atalla architecture, based on the ppci (Pure Python Compiler Infrastructure) framework. Compiles AtallaC (C with vector/mask types and hardware intrinsics) to Atalla assembly.

### 3.1 AtallaC Language Extensions

**Types and built-ins:**

| Feature | Syntax | Purpose |
|---------|--------|---------|
| Vector type | `vec v;` | 32-element vector register |
| Mask type | `mask m;` | Predication mask |
| GEMM intrinsic | `gemm(a_row, c_row, mask)` | 32├Ч32 systolic array multiply-accumulate |
| Vector op | `vec_op_masked("+", a, b, mask)` | Masked elementwise operation (+, -, *, RMAX, RSUM, EXP) |
| Make mask | `make_mask(">", a, b, mask)` | Compare vectors, produce mask |
| Lane extract | `float x = v[i];` | Extract scalar from vector lane |
| Inline asm | `asm("lw_s %0, 0(%1)" : "=r"(out) : "r"(addr))` | Direct assembly insertion |

**First-class intrinsics (from PR #39):**

| Intrinsic | Signature | Replaces |
|-----------|-----------|----------|
| `scpad_load` | `scpad_load(sp, gmem_addr, sdma_ctl)` | `asm("scpad_ld %0, %1, %2" ...)` |
| `scpad_store` | `scpad_store(sp, gmem_addr, sdma_ctl)` | `asm("scpad_st %0, %1, %2" ...)` |
| `vector_load` | `v = vector_load(addr, rs2, num_cols, sid)` | `asm("vreg_ld %0, %1, %2, N, S" ...)` |
| `vector_store` | `vector_store(v, addr, rs2, num_cols, sid)` | `asm("vreg_st %0, %1, %2, N, S" ...)` |
| `load_weights` | `load_weights(wvec)` | `asm("lw_vi %0, %1, 0, m0" ...)` |
| `sqrt` | `float y = sqrt(x)` | Scalar BF16 square root |

**Implicit division:** `float inv = 1.0 / x;` compiles to `rcp_bf + mul_bf`. Type casts (`float(x)`, `(int)y`) emit `stbf_s`/`bfts_s` implicitly. No explicit inline asm needed for reciprocal/conversion.

**Still require inline asm (no intrinsic):** `lw_s` (scalar config loads), `li_s` (large immediates), `halt`.

### 3.2 Compilation Flow

```
AtallaC source (.c)
    тФВ  c_emitter.py generates parameterized source
    тЦ╝
ppci atalla_cc frontend (lexer + parser)
    тФВ  optimization passes
    тЦ╝
ppci IR (SSA form) тЖТ instruction selection тЖТ register allocation
    тЦ╝
Raw assembly output (.s)
    тФВ  build_compiler.compile_asm():
    тФВ    parse_program тЖТ inject_main_bootstrap
    тФВ    тЖТ expand_large_li (li_s тЖТ lui_s + addi_s)
    тФВ    тЖТ schedule_program тЖТ relocate_and_encode
    тЦ╝
Encoded instruction packets (.in format)
```

### 3.3 What the Compiler Can and Cannot Do

**Works well (actively used in pipeline):**
- Scalar control flow (while loops, conditionals)
- `gemm()` intrinsic тЖТ emits `gemm.vv`
- `vec_op_masked()` for element-wise ReLU, Softmax, LayerNorm kernels
- First-class intrinsics: `scpad_load/store`, `vector_load/store`, `load_weights`, `sqrt`
- Implicit division: `1.0 / x` тЖТ `rcp_bf + mul_bf`; type casts handle `stbf/bfts`
- Inline asm for `lw_s`, `li_s`, `halt` (no intrinsics for these yet)
- Large constant loading via `li_s` тЖТ expanded by `build_compiler` to `lui_s` + `addi_s`

**Known limitations:**
- **Vector spill / stack frame overflow**: When the register allocator spills many 64-byte vectors to the stack, offsets can exceed the 12-bit signed immediate range (┬▒2048 bytes), causing instruction selection failure. Workaround: restructure code to reduce live vector count (reuse variables, compute+store immediately). LayerNorm required this restructuring.
- **No vector SQRT**: `sqrti_vi` is commented out in the ISA defs. Only scalar `sqrt()` intrinsic works. Vector sqrt must be decomposed to scalar lane extraction + scalar sqrt (acceptable when only one scalar result is needed, e.g., LayerNorm variance).
- **No `shift.vi`**: Prevents on-chip horizontal stride-select for MaxPool.
- **No pointer dereference for MMIO**: `*(volatile int*)0x3C` doesn't work; all MMIO reads require inline asm.

### 3.4 Current Role in Pipeline

The compiler is the **primary compilation path** for all emulated compute kernels. The kernel generators in `atalla-graph/kernels/` produce AtallaC using first-class intrinsics (`scpad_load/store`, `vector_load/store`, `load_weights`, `sqrt`). The compiler handles scalar control flow, intrinsic lowering, register allocation, and instruction selection. Only config loads (`lw_s`), large immediates (`li_s`), and `halt` remain as inline assembly.

**Reference .c kernels that compile successfully:** relu, softmax, maxpool, gemm_tiled, conv, layernorm (all 6). LayerNorm required restructuring to reduce peak live vector count below the stack frame offset limit.

## 4. Component Deep Dive: functional_sim

**Purpose:** Functional simulator for the Atalla accelerator plus the assembly/encoding toolchain that produces `.in` files.

### 4.0 Encoding Toolchain (build.py vs build_compiler.py)

Two encoding systems exist, each serving a different purpose:

**`build.py` (Sahil's original assembler):**
- `encode_instruction(dict)` тАФ encodes a single instruction dict to 48-bit hex
- `assemble_file(asm_str)` тАФ parses `.s` text, encodes each line via `encode_instruction`
- `emit_test_format(instrs)` тАФ packs into VLIW packets (sequential, NOT dependency-aware)
- `DRAMWriter` тАФ builds data memory sections (config, tensors)
- `render_testfile(instr_text, dram_text)` тАФ combines instruction + data sections into `.in`
- Used by: hand-written `build_*.py` scripts

**`build_compiler.py` (VLIW scheduler, moved from `emulator/`):**
- `compile_asm(asm_str)` тАФ full pipeline: parse тЖТ bootstrap тЖТ li_s expansion тЖТ **dependency-aware VLIW scheduling** тЖТ encode via `_asm_encoding.py` тЖТ `.in` instruction section
- Handles ppci register naming (`xN`/`vN`/`mN`), compiler directives, branch targets
- Enforces VLIW constraints: max 1 vector op, max 1 memory op per packet
- Latency-aware scheduling from `instruction_latency.py`
- Used by: `compile_and_convert.py`, `atalla-graph` pipeline

**Key difference:** `build.py`'s `emit_test_format` puts 1 instruction per packet (VIRTUAL_PACKET_SIZE=1 by default) тАФ safe but wastes 75% of slots. `build_compiler.py` does real dependency analysis to pack multiple independent instructions per packet.

**Both produce the same `.in` format.** The instruction section is hex-encoded VLIW packets; the data section is address-value pairs. `render_testfile()` from `build.py` combines them.

### 4.1 Architecture Modeled

| Component | Python Class | Description |
|-----------|-------------|-------------|
| Scalar register file | `ScalarRegisterFile` (34 regs) | General-purpose 32-bit registers; x0=zero, x2=stack pointer, x33=vector spill base |
| Mask register file | `ScalarRegisterFile` (16 regs) | Predication masks; m0 hardwired to all-ones (0xFFFFFFFF) |
| Vector register file | `VectorRegisterFile` | 64 vector registers, each 32 elements of bf16 |
| Scratchpad banks | `Scratchpad` ├Ч 2 (SP0, SP1) | 32-slot on-chip SRAM for tiles, each slot = 32 bf16 values |
| Execute unit | `ExecuteUnit` | Scalar ALU, vector ALU, GEMM unit, control |
| Performance metrics | `PerfMetrics` | Counters for cycles, instructions, GEMM ops, SDMA ops, mem ops |
| Global memory | `Memory` | Main DRAM, dictionary-based, 32-bit words at 2-byte stride |

### 4.2 `.in` File Format

The `.in` file is a text file with two sections:

```
# Instruction memory (encoded instruction packets, 4 bytes each)
0x12345678
0xABCDEF01
...
0x00000000   # end marker

# Data memory (addr: value pairs, 2-byte stride addresses)
@0x003C 0x00001000
@0x1000 0x3F800000
...
```

Each data entry is a 32-bit word at a 2-byte-stride address. Two BF16 values are packed per word by `DRAMWriter.to_u32_words()`. The emulator's `Memory.read_data(addr)` returns the 32-bit word at the given address; `read_bf16_from_memory()` in `run_model.py` extracts the low 16 bits as a BF16 value.

### 4.3 Emulator Execution

```python
mem = Memory(in_file)       # load .in file into instruction + data memory
run_emulator(mem, sregs, mregs, vregs, SP0, SP1, EU, pc=0, issue_width=4, ...)
```

The emulator:
1. Fetches the instruction word at PC
2. Decodes opcode, extracts fields
3. Executes (scalar ALU, vector ALU, memory, GEMM, control)
4. Updates performance counters (cycles, instruction count, GEMM ops, SDMA ops, mem ops)
5. Advances PC (or branches)
6. Repeats until `halt.s`
7. Dumps state to output files (memory, registers, scratchpad, perf metrics)

### 4.4 `DRAMWriter`

A helper that builds the data memory section:

```python
img = DRAMWriter()
img.u32(addr, value)      # write 32-bit integer at byte address
img.bf16(addr, float_val) # convert float to bf16, write 16-bit at byte address
data_text = img.render_data_mem(include_zeros=True)
```

Internally, `bf16(addr, x)` stores a 16-bit BF16 value. `to_u32_words(stride=2)` packs two consecutive BF16 values into each 32-bit word.

### 4.5 ADDR_TABLE Convention

All kernels read their parameters from a fixed address table at byte address **60 (0x3C)**. The table layout varies per kernel:

**GEMM/Conv/Linear:**

| Offset | Field |
|--------|-------|
| +0 | A_GMEM (input matrix address) |
| +4 | W_GMEM (weight matrix address) |
| +8 | C_GMEM (output matrix address) |
| +12 | M (rows) |
| +16 | N (cols) |
| +20 | K (inner dimension) |
| +24 | M_tiles |
| +28 | N_tiles |
| +32 | K_tiles |
| +36 | TILE (tile size, always 32) |

**ReLU:**

| Offset | Field |
|--------|-------|
| +0 | IN_GMEM |
| +4 | OUT_GMEM |

**Softmax:** Same as ReLU (IN_GMEM at +0; +4 unused).

### 4.6 Critical Emulator Fixes

Several bugs were discovered and fixed during pipeline integration:

**1. Mask register m0 hardwired to all-ones:**

The Atalla hardware convention is that `m0` is always all-ones (0xFFFFFFFF), used as the default "no masking" predicate. The emulator's `ScalarRegisterFile` returns 0 for register 0 by default. Fix: patch `mregs.read` at emulator startup to return `0xFFFFFFFF` for index 0.

**2. `gemm.vv` computation order:**

The `lw.vi` instruction loads weight rows into the systolic array's internal buffer as *columns*, so `gemm_weights` represents W^T. The correct multiply is `gemm_weights @ v_in + v_acc` (i.e., W^T @ v_in). This was corrected from the original which had the operands in the wrong order.

**3. `lw.vi` weight buffer reset:**

In multi-tile GEMM (e.g., AlexNet conv1 with 32 tiles), the `gemm_weights` matrix must be reset to zero before each new weight loading phase. Without this, stale weights from the previous tile corrupt subsequent tiles. Fix: track a `_gemm_ran` flag; when `lw.vi` is encountered after a `gemm.vv`, reset `gemm_weights` and `num_weights` to zero.

**4. Stack pointer / DRAM overlap:**

The ppci compiler uses scalar register `x2` as a stack pointer for local variables. It was initially set to `0x8000`, but for large models (e.g., AlexNet conv1 with 1024├Ч27 input matrix), the stack grows downward into the `A_GMEM` data region (starting at `0x1000`). Compiler-generated `sw.s` instructions for loop counters overwrote input matrix data, causing NaN and huge value propagation. Fix: compute stack base dynamically as `((max_data_addr + 0x1000) & ~0xFFF) + 0x1000`, placing it safely above all DRAM data. The vector spill base (`x33`) is set to the same value.

## 5. Component Deep Dive: Kernel Library

### 5.0 Two-Layer Kernel Truth

The kernel library exists at two levels:

| Level | Location | Format | Purpose |
|-------|----------|--------|---------|
| **Lowest truth** | `functional_sim/build_*.py` | Hand-written asm + data | Golden reference: directly encodes tested instructions + generates complete `.in` with DRAMWriter data sections. Runs on emulator, validated against PyTorch. |
| **Reference C** | `aihw-ppci-compiler/atalla_tests/kernels/*.c` | AtallaC with intrinsics | Canonical C implementations using `scpad_load/store`, `vector_load/store`, `load_weights`, `sqrt()`, `1.0/x`. Compile via ppci тЖТ `.s` тЖТ `build_compiler` тЖТ `.in`. |
| **Code generators** | `atalla-graph/kernels/*.py` | Python тЖТ AtallaC strings | Parameterized generators called by `c_emitter.py`. Must structurally match the reference `.c` kernels. |

**Validation flows bottom-up:** build_*.py (asm) тЖТ reference .c (compiled) тЖТ atalla-graph generators (parameterized). Each level must match the one below.

The `functional_sim/build_*.py` files contain hand-written assembly generators. These are **not used by the pipeline's graph operations** (which use the C compiler path), but serve as the **lowest truth** for standalone validation and benchmarking. Each script generates a complete `.in` file with both instructions and DRAM data.

### 5.1 `make_tiled_gemm_asm(M, N, K)` тАФ Tiled GEMM

Used for: **standalone GEMM testing** (pipeline uses `_gemm_c()` instead)

Algorithm:
```
for mi in 0..M_tiles:
  for ni in 0..N_tiles:
    scpad.ld C_tile from DRAM       # load partial output tile
    for ki in 0..K_tiles:
      scpad.ld A_tile from DRAM     # load input tile to SP0
      scpad.ld W_tile from DRAM     # load weight tile to SP1
      for w in 0..TILE:
        lw.vi w                     # preload weight row w into systolic array
      for r in 0..TILE:
        vreg.ld a_row from SP0      # load input row
        vreg.ld c_row from SP1      # load accumulator row
        gemm.vv c_row, a_row, mask  # systolic MAC: c += a * W
        vreg.st c_row to SP1        # store updated accumulator
    scpad.st C_tile to DRAM         # write output tile back
halt
```

Key instructions:
- `lw.vi` тАФ preloads one weight row into the systolic array's internal buffer (32 bf16 values)
- `gemm.vv` тАФ performs one row of multiply-accumulate: `C[r] += A[r] * W_buffer`
- `scpad.ld/st` тАФ bulk DMA between DRAM and scratchpad
- `vreg.ld/st` тАФ move single rows between scratchpad and vector registers

### 5.2 `make_relu_asm(total, width)` тАФ ReLU

Uses compare-and-blend: `mgt.mvv mask, v, zero` тЖТ `sub.vv result, result, result` тЖТ `add.vv result, result, v, mask` (only positive lanes get copied).

### 5.3 `make_softmax_asm(length)` тАФ Softmax

Uses hardware reduce-max, reduce-sum, exp, and bf16 reciprocal instructions: `rmax тЖТ sub тЖТ exp тЖТ rsum тЖТ stbf_s тЖТ rcp_bf тЖТ bfts_s тЖТ mul.vs`.

### 5.4 `make_maxpool_asm(H_in, W_in, pool_size, stride)` тАФ MaxPool

Single-channel MaxPool operating on one H├ЧW tile using pairwise `mgt` + blend for both vertical and horizontal max operations.

Limitation: Single-channel only (W тЙд 32 elements). Multi-channel requires a per-channel loop wrapper not yet wired into the pipeline.

### 5.5 `make_attention_asm(S, d)` тАФ Self-Attention

Full Q┬╖K^T тЖТ softmax тЖТ ┬╖V attention in one kernel. Uses GEMM for matrix products, inline softmax for score normalization, and a second GEMM for the output.

## 6. Full Pipeline Integration

### 6.1 Entry Point: `run_model.py`

```python
# Phase 1: Capture
gm = capture(model, example_input)         # FX trace + shape prop + op normalization
gm = remove_ops(gm)                        # BN fold, dropout removal
gm = plan_tiles(gm)                        # tile planning + DRAM layout

# Phase 2: Execute layer by layer
activation_cache = {}
kernel_metrics = []
for node in gm.graph.nodes:
    if node.op in ("placeholder", "get_attr"):
        activation_cache[node.name] = ...   # store input/weights
        continue
    if atalla_op in ("flatten", "dropout"):
        activation_cache[node.name] = reshape(prev)
        continue

    emission = emit_node(node, gm, activation_cache)

    if emission.skip_emulator:               # NumPy fallback (maxpool, add, mul)
        activation_cache[node.name] = emission.numpy_result
    else:                                    # C compiler + emulator
        compile_and_assemble(emission, ...)   # ppci compile C тЖТ asm тЖТ schedule тЖТ encode
        in_file = render_in_file(emission)    # combine encoded instrs + DRAM data тЖТ .in
        write(in_file)
        mem, eu = run_on_emulator(in_file)    # execute on simulator
        result = read_bf16_from_memory(mem, emission.output_addr, emission.output_elements)
        activation_cache[node.name] = result
        # collect perf metrics from eu.perf_metrics

# Phase 3: Validate
compare(activation_cache[final_node], pytorch_reference)
```

### 6.2 Layer-to-Layer Data Passing

Each layer is executed as a **standalone emulator invocation**:
- The emulator starts fresh for each layer (registers zeroed, scratchpad empty)
- The stack pointer (`x2`) is dynamically computed above all DRAM data for each layer
- Input activations from the previous layer's output are serialized into the new layer's DRAM image
- The Python orchestrator reads the output from emulator memory and passes it to the next layer

This is equivalent to a "layer-at-a-time" execution model.

### 6.3 The C Compiler Path (`compile_and_assemble`)

```python
def compile_and_assemble(emission, work_dir, tag):
    if emission.c_source:
        raw_s = compile_c(emission.c_source, work_dir, tag)  # ppci atalla_cc тЖТ .s
        in_content, _, _ = build_compiler.compile_asm(raw_s)  # schedule + encode
        emission.instr_text = in_content.split("\n.data")[0].strip()
    return emission.instr_text
```

`build_compiler.compile_asm()` performs:
1. `parse_program()` тАФ parse assembly into instruction list + label map
2. `inject_main_bootstrap()` тАФ add startup code
3. `expand_large_li()` тАФ expand `li_s` pseudo-instructions to `lui_s` + `addi_s`
4. `schedule_program()` тАФ VLIW scheduling into instruction packets
5. `relocate_and_encode()` тАФ resolve labels, encode 32-bit instruction words
6. `emit_packet_format()` тАФ format as hex lines for `.in` file

## 8. Performance Metrics

### 8.0 Measuring Kernel Efficiency Without a Cycle-Accurate Simulator

The functional simulator is **not** cycle-accurate тАФ it executes one packet per step without modeling pipeline stalls, cache misses, or memory latency. However, several meaningful efficiency metrics are available:

| Metric | Formula / Source | What it tells you |
|--------|-----------------|-------------------|
| **VLIW slot utilization** | `instructions / (packets ├Ч 4)` | ILP quality. 1.0 = all 4 slots filled. Higher = better scheduling. |
| **Packet count** | `perf_metrics["packets"]` | Lower bound on cycles (1 packet = 1 issue). Primary code size metric. |
| **Instruction count** | `perf_metrics["instructions"]` | Total non-NOP instructions executed. |
| **NOP ratio** | `1 - slot_utilization` | Wasted slots from scheduling constraints. |
| **Arithmetic intensity** | `flops_total / bytes_loaded` | Compute vs memory bound. Higher = more compute per byte moved. |
| **SDMA/vector/scalar mix** | From instruction breakdown | Identifies bottleneck (memory-bound vs compute-bound). |
| **GEMM ops** | `perf_metrics["gemm_ops"]` | Systolic array utilization count. |

**Slot utilization is the primary efficiency metric** in the absence of a cycle-accurate sim. It measures how well the VLIW scheduler (`build_compiler.py`) packs instructions. Theoretical maximum is 4 instructions/packet, but VLIW constraints (max 1 vector + max 1 memory per packet) and data dependencies limit this.

**Comparing hand-written vs compiled kernels:** Run both through the emulator and compare packet count (compiled should be within ~1.5├Ч of hand-written) and slot utilization (compiled should be > 0.5).

The existing `benchmark.py` generates 19 graphs including packet utilization (graph 17), cycle ranking (graph 16), memory traffic (graph 18), and arithmetic intensity.

### 8.3 Analysis

**BasicModule:** Individual kernels achieve near-perfect cosine similarity (1.0) against PyTorch float32 reference. The final output (cos=0.868) shows expected BF16 accumulation drift compounding through 3 GEMM layers.

**AlexNet:** BF16 precision errors compound through 19 execution layers. Conv1 alone has cos=0.16 vs fp32 reference due to the large im2col matrix (1024├Ч27 тЖТ 32 tiles), where each tile's BF16-truncated intermediate results accumulate quantization error. Later layers see progressively lower cosine similarity as errors cascade. This is a fundamental BF16 precision limitation тАФ the emulator is functionally correct (no NaN, no out-of-range values), but the hardware's 16-bit mantissa cannot match float32 fidelity across deep networks.

**Cycle distribution (AlexNet):**
- Conv1 dominates (120K / 183K = 65% of total cycles) due to 1024 GEMM operations over 32 tiles
- Each GEMM kernel invocation costs ~1,700 cycles (consistent with BasicModule's matmul)
- ReLU is cheap: ~190 cycles per invocation regardless of element count

### 9.2 Design Decisions

| Decision | Rationale |
|----------|-----------|
| **C compiler + intrinsics for kernels** | Compiler handles scalar control flow + intrinsic lowering; only `lw_s`/`li_s`/`halt` remain as inline asm |
| **Per-layer emulator invocation** | Emulator designed for single-kernel execution; fresh state per layer is cleanest |
| **Dynamic stack pointer** | Prevents compiler stack frames from overlapping DRAM data regions |
| **BF16 weight conversion at capture time** | Matches hardware precision; avoids surprise fp32тЖТbf16 truncation mid-pipeline |
| **im2col for convolutions** | Reduces Conv2d to matrix multiply, reusing the tiled GEMM kernel |
| **MaxPool in NumPy** | Kernel exists but is single-channel; multi-channel wrapper not yet wired |
| **Add/Mul in NumPy** | Bias adds and residual scales; AlexNet folds bias into Conv/Linear |
| **build_*.py as lowest truth** | Hand-written asm bypasses compiler; validates emulator + encoding independently |
| **atalla-graph owns data sections** | `c_emitter.py` + `DRAMWriter` generate config + tensor DRAM layout for each graph node |

### 9.3 Known Limitations

| Limitation | Impact | Mitigation |
|------------|--------|------------|
| **Compiler stack frame overflow** | Kernels with many live vectors exceed 12-bit offset | Restructure code to reduce peak live vectors (layernorm required this) |
| **No vector SQRT / shift.vi** | Blocks full on-chip MaxPool and vector-width LayerNorm | Use scalar sqrt intrinsic; MaxPool vertical-only on chip |
| **MaxPool uses NumPy** | Not running on emulator; no cycle counts for maxpool | Wire `build_maxpool.py` per-channel loop |
| **Emulator stbf/bfts bug** | Softmax reciprocal produces wrong values in emulator | Tracked; use `1.0/x` in C which compiles correctly |
| **BF16 accumulation drift** | AlexNet cos=0.20 vs fp32 after 19 layers | Expected; tracked via per-kernel cosine similarity |
| **Single-batch only** | Batch size > 1 not tested | Tile planner assumes batch=1 |
| **ADDR_TABLE at fixed address 60** | All kernels share the same parameter address | Works for layer-at-a-time |
| **Bias not fused into GEMM** | Conv/Linear bias requires separate NumPy add | Fused GEMM+bias kernel would be more efficient |

### 10.1 Validation Harness (next priority)

A unified validate-and-benchmark harness for each kernel, working bottom-up:

**Step 1 тАФ Validate build_*.py (hand-written asm, lowest truth):**
For each kernel: run the `build_*.py` script тЖТ generate complete `.in` тЖТ run emulator тЖТ extract output from scratchpad/DRAM тЖТ compare against PyTorch reference (cosine similarity, max error). Capture metrics (packets, instructions, slot utilization). These are the golden outputs.

**Step 2 тАФ Validate compiled .c kernels against golden:**
Compile `atalla_tests/kernels/*.c` тЖТ `.s` тЖТ `build_compiler.compile_asm()` тЖТ instruction section. Reuse the same `DRAMWriter` data section from Step 1 (same config, same input tensors). Run emulator. Compare outputs bit-for-bit against Step 1's golden outputs. Compare metrics: does the compiler version use more packets? Lower slot utilization?

**Step 3 тАФ Validate atalla-graph generators:**
Call `atalla-graph/kernels/*.py` generators with the same dimensions тЖТ produce AtallaC тЖТ compile тЖТ same comparison. This validates that the parameterized generators produce structurally correct code.

**Data section generation** is owned by `atalla-graph` for the graph pipeline (`c_emitter.py` + `dram_builder.py` + `DRAMWriter`). For standalone kernel validation (Steps 1-2), the existing `build_*.py` scripts and `benchmark.py` already generate complete data sections with `DRAMWriter`.

### 10.2 Remaining Pipeline Gaps

1. **Wire maxpool to emulator**: `build_maxpool.py` works for single-channel tiles. Need a per-channel loop in `emit_maxpool()` that runs the kernel once per channel with appropriate DRAM offsets.

2. **Compiler stack frame overflow**: Large kernels (many live vectors) exceed the 12-bit signed offset limit for scratchpad-relative addressing. Workaround: restructure code to reduce peak live vectors. Long-term fix: add large-offset addressing patterns to ppci backend.

3. **BF16 precision drift**: Conv1's cos=0.16 against fp32 suggests 32-tile accumulation magnifies BF16 truncation. Investigate fp32 accumulator with bf16 operands.

4. **Bias fusion**: Add fused GEMM+bias kernel to eliminate separate NumPy add pass.

5. **Emulator `stbf.s`/`bfts.s` bug**: Softmax produces all-zeros from the emulator due to incorrect floatтЖФinteger bit pattern conversion in the functional sim. Tracked as a known emulator bug.

6. **Missing ISA ops**: `shift.vi` (horizontal stride-select), `sqrti.vi` (vector sqrt) are commented out in ISA defs. Blocks full on-chip MaxPool and vector-width LayerNorm.
